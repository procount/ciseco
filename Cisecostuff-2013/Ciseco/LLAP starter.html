LLAP Protocol

LLAP is quite different to most message protocols (the word protocol simply means a predefined rule).

A first glance LLAP uses a simple, short, text based message, to describe events (or data) in a relatively "human" way. It was designed to be easily used and understood. Just as HTTP gave a way to link pages on the web, LLAP provides a simple way to exchange messages between low cost electronic devices.

The message exchanged does not describe how to get the actual content "there", this is very significant. By doing so, the emphasis is on the systems underneath (e.g. wireless card or gateway) is to get the data there "totally transparently".

The benefit is three fold

  • It makes things far simpler for the user (cost, time and experience required are all significantly reduced)
  • LLAP can be transmitted by any communications medium (future proof)
  • LLAP messages can be exchanged between the very simplest of devices (cost)

The message is of a fixed length which makes wireless transmission much more efficient and manageable. It's 12 characters can be compressed to just 8 if needed. This is useful because there are many other systems which communicate in 8 byte chunks (LIN & ANT being just two) or those of very limited speed (eg InfraRed).

Any medium that can transport clear text can support LLAP,  essentially from a serial port to TCP/IP(internet) and anything else in-between. This means it's future proof, flexible and inexpensive. So inexpensive and requiring such little processing power that LLAP can be run on hardware costing less than a can of coke.

The simplicity of LLAP belies the 5 years of work and testing behind it. What's more it's free and always will be.

 

Sending and receiving a simple example

The 12 characters (lets call them bytes) describe the following 3 things

  • Where the message starts (byte 1)
  • What/who should take notice of this data(bytes 2 & 3)
  • The message (or data) we'd like to exchange (bytes 4 to 12)

The HELLO command is probably the most basic of examples, it's like a PING on a computer or a HI between people. We send out essentially "hey device AA are you there?" here's what we broadcast (send)

aAAHELLO----

Let's see what this message means. A start byte of "a" shows where all the devices need to start listening, "AA" shows we are asking the device numbered AA to listen to us (everyone else can ignore the data) and the "HELLO----" is the data we are sending. The end stream of dashes is there to "pad out" the message to 12 bytes and no more.

Device AA sends back a similar response of.

aAAHELLO----

Entering LLAP commands via a terminal window

LLAP commands must be sent in one radio packet. The best way to achieve this is to enter a command into a text editor and then copy/paste into the terminal window. Typing is too slow as the radio will send the packet as soon as the ATRO timeout has occured - this is usually set to 16mS.. Using XCM the commands can can be entered into the text bar and then the send button can be pressed.

Sending and receiving a temperature

Lets imagine the second device on our network is device "AB" and it's a temperature sensor. The person or company who built it, chose to use the word "TEMP" to denote we want to know the temperature. With this fact in hand, if we send

aABTEMP-----

we may receive a response like this

aABTEMP+23.5

Network methodology

A network theoretically can have 1024 devices, however using A to Z and A to Z for the device ID's gives a potential of 676 and space for groups and specialist methods like device to master initiated transactions.

Transmissions should be no shorter apart than 100ms. This allows for plenty of "dead time" to repeat messages and receive uninitiated broadcasts from sleeping devices. Using this method we can accurately calculate and predict the performance of various transmission types (wireless, IP, serial etc).

The transit time is expected to be below the accepted human perceived level of 250ms, this gives the appearance of the network running in "real time" with little visible evidence of "lag" as seen in other commercial systems.

Summary

LLAP is a balanced compromise between humans and machines. It can be used on almost any hardware. Performance, cost, ease of use and flexibility are paramount. When used wirelessly, it's aim is to achieve the "last few meters" and not to try and emulate a wired network. Such "routing" topologies when used for "device networks" invite significant overhead and complexity often for little gain. LLAP demonstrates that a simple idea can deliver real functionality, reduced cost and in a "human" way. A LLAP network is ideally suited to low power, even battery powered devices, yet has huge scalability through the use of gateways. For example 676 devices can share a single TCP/IP address, multiply that by the amount of IP addresses and it's possible to build networks of many, many millions of devices.
